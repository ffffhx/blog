# 技术博客文章写作 Skill

这个 skill 用于当前仓库里的技术博客写作流程。

当用户有下面这些需求时使用它：

- 分析一个代码仓库并整理成文章
- 给这个博客写一篇技术分享
- 在 `source/_posts` 下新增或修改源码解析类文章
- 为文章生成示意图和本地资源文件

下面这些情况不要用它：

- 不会落成博客文章的短问答
- 非技术类随笔，除非用户明确要求沿用同样流程

## 当前仓库的内容约定

开始写之前，先读：

1. `README.md`
2. `source/_posts` 里最近的一到两篇技术文章

当前仓库现在是一个 `Next.js + React + TypeScript + Tailwind CSS` 博客，但文章内容依然保存在 Markdown 文件里，路径是：

- `source/_posts/YYYY/MM/DD/*.md`
- `source/about/index.md`

这个仓库里和文章相关的兼容规则是：

- 保留 `title`、`date`、`categories`、`tags`、`excerpt` 这些 front matter 字段
- 文章资源放在和文章 basename 同名的同级目录里
- 本仓库仍然兼容 `{% asset_img figure-01.svg %}` 这种标签，文章本地资源优先使用这种写法
- 最终用 `pnpm build` 验证文章渲染是否正常

## 默认工作流

除非用户明确要求缩小范围，否则按这个顺序执行：

1. 先确认文章目标。
   判断这是新文章还是改旧文章。如果是分析外部仓库，要在正文里固定观察时的 commit 和观察日期。

2. 先看目标材料，再动笔。
   如果是仓库分析，至少要看 README、入口文件、构建文件、核心模块分层、鉴权层、输出层、测试层，以及任何定义架构边界的文件。没有形成清晰模块图之前，不要开始写正文。

3. 再看当前博客仓库的写作约定。
   阅读 `README.md` 和最近的一到两篇技术文章，保证语气、front matter、章节密度和资源引用方式与现有博客一致。

4. 先搭文章结构，再写正文。
   优先使用下面这个顺序：
   - 摘要
   - 术语解释 / 阅读预备
   - 问题定义或产品定位
   - 主要架构章节
   - 关键代码与设计取舍
   - 工程化 / 测试 / 发布
   - 最后总结

5. 直接在仓库里起稿。
   把 Markdown 文件放进 `source/_posts/YYYY/MM/DD/`。同时创建同名资源目录，避免后面补图时再改路径。

6. 按章节补示意图。
   每个主要章节至少配一张图。优先使用带中文标注的 SVG。图尽量简单，用框、箭头和短说明就够，不要做装饰性复杂设计。

7. 正文结构稳定后再补代码片段。
   文章里的代码必须满足：
   - 只保留表达设计意图所需的最小逻辑
   - 必要时明确说明这是裁剪版或改写版片段
   - 每一行都加中文注释
   - 能回链到真实源码时尽量回链

8. 本地验证。
   运行 `pnpm build`。如果是仓库里与本次文章无关的问题导致失败，要明确说明；否则不要在验证前结束。

如果需要更短的执行清单和文章骨架，继续读：

- `references/workflow-checklist.md`
- `references/article-template.md`

## 新文章创建规则

当新建技术文章时：

- 文件放在 `source/_posts/YYYY/MM/DD/`
- 文件名优先使用稳定、可读、描述性强的中文标题或与仓库风格一致的 slug
- 同时创建与文章 basename 同名的资源目录

如果用户没有指定文件名，就从文章标题推导，保持可读性。

不要改动无关文章和资源。

## 写作规则

优先使用直接、清楚、平实的中文。

默认读者可能并不了解目标语言、框架或系统，所以要：

- 尽早解释不常见术语
- 在讲文件之前，先讲清楚为什么会有这一层
- 每段只讲一个核心点
- 不要把边角细节硬包装成重要设计

如果是源码解析文章：

- 先从外到内解释这个系统
- 再把代码目录和职责对应起来
- 然后再放大到最关键的执行链路
- 最后落到 tradeoff 和可学习点

## 代码片段规则

文章里的代码片段不是原样转储。

必须遵守这些规则：

- 把代码裁到刚好能表达核心意思的程度
- 保持语法至少基本可读
- 每一行都加中文注释，包括 import、花括号和结束行
- 解释负担尽量放在行内注释里，不要只放在正文外层

推荐写法：

```go
func main() { // 程序入口
    run() // 把控制权交给核心流程
} // 入口结束
```

不推荐的写法：

- 直接扔一段 60 行原始代码不裁剪
- 中英注释混杂且没有阅读节奏
- 只挑几行注释，剩下让读者自己猜

## 示意图规则

除非用户明确要求别的格式，否则优先使用 SVG。

默认约定：

- 尺寸大约 `1200 x 760`
- 中文标注
- 中性偏暖背景
- 2 到 4 种有分组语义的颜色
- 图内自带简短标题和副标题

图片文件名按下面这种顺序编号：

- `figure-01.svg`
- `figure-02.svg`
- ...

在 Markdown 里用下面这种方式引用：

```md
{% asset_img figure-01.svg %}
```

## 验证规则

结束前必须确认：

1. 文章文件在正确的 `source/_posts/...` 路径下
2. 资源目录存在，图的引用路径正确
3. 运行 `pnpm build`
4. 总结这次创建了什么，以及是否还有遗留注意项

## 这个 Skill 沉淀了这次写作流程的哪些经验

这个 skill 基于本仓库里一次源码解析文章的完整流程，总结了下面这些默认顺序：

1. 写之前先调研目标仓库
2. 起稿前先看博客仓库自己的写作约定
3. 先用读者友好的顺序搭好文章骨架
4. 直接把文章写进 `source/_posts`
5. 为每个主要章节创建中文 SVG 示意图
6. 对代码示例做强裁剪，并逐行写中文注释
7. 最后用本地站点构建做验证

除非用户明确要求轻量版本，否则这就是默认执行顺序。
